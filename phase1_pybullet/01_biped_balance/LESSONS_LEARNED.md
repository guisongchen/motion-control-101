# 双足站立控制 - 经验教训记录

**日期**: 2026-04-06  
**项目**: Phase 1.1 - PyBullet 2D双足站稳  
**状态**: 进行中 - 控制策略需改进

---

## 项目目标

让双足机器人在PyBullet仿真中稳定站立5秒以上，能抵抗小扰动。

---

## 尝试过的控制策略

### 1. 基础PD关节控制 (biped_balance.py)

**方法**:
- 为每个关节独立实现PD控制器
- 根据 torso 倾斜角度调整目标关节角度
- 使用 Torque Control 模式

**结果**: ❌ 失败
- 机器人在 2.7秒 左右摔倒
- 仅控制关节角度，没有考虑整体质心位置

**问题分析**:
- 没有显式控制质心(CoM)在支撑多边形内
- 各关节控制独立，缺乏协调
- 脚踝力矩不足以快速纠正倾斜

---

### 2. 分层控制架构 (biped_balance_v2.py)

**方法**:
- 分层控制: 质心控制 → 关节分配
- 高度控制: 通过膝盖屈伸保持质心高度
- 姿态控制: 通过踝关节调整保持直立
- 使用 Position Control 模式

**结果**: ❌ 数值不稳定 (NaN)
- 控制增益设置过高导致数值溢出
- 即使降低增益，机器人仍有前倾趋势

**问题分析**:
- Position Control 的 `positionGain` 参数与PD增益混淆
- 初始姿态不佳 (落地后 torso 有 5-20° 前倾)
- 高度控制和姿态控制耦合导致冲突

---

### 3. 简化PD控制 (biped_balance_simple.py)

**方法**:
- 简化为三个主要控制量:
  - 高度 → 膝盖弯曲
  - 姿态 → 脚踝角度
  - 位置 → 髋关节补偿
- 使用保守的PD增益

**结果**: ❌ 数值不稳定
- 即使增益很低，仍在 1秒内出现 NaN
- PyBullet 的数值积分器在特定条件下不稳定

**问题分析**:
- Humanoid 模型的关节约束和惯性参数导致数值敏感
- 脚与地面接触时的冲击力未被妥善处理

---

### 4. 纯力矩控制 (biped_balance_stable.py)

**方法**:
- 使用 Torque Control 模式
- 显式设置物理引擎参数 (numSolverIterations, numSubSteps)
- 设置初始站立姿态 (resetJointState)

**结果**: ⚠️ 部分有效但不稳定
- 落地后高度约 0.82m
- 但 torso 仍有 20° 左右倾斜
- 控制启动后 0.4秒 内摔倒

**问题分析**:
- 初始姿态设置后，物理仿真会立即改变关节角度
- 力矩控制响应慢，无法快速纠正大角度误差
- 缺乏对关节角速度的阻尼

---

### 5. 协调关节控制 (biped_balance_working.py)

**方法**:
- 同时控制 hip, knee, ankle 协调运动
- 姿态控制优先 (脚踝纠正倾斜)
- 高度控制限制范围 (避免过度弯曲)

**结果**: ❌ 机器人"起飞"
- 高度从 0.88m 增加到 2.7m
- 膝盖控制逻辑错误导致向上加速

**问题分析**:
- 高度误差计算: `h_err = target - current`
  - 当 height < target 时，h_err > 0
  - 膝盖弯曲为正角度，应该产生向上的力
  - 但实际效果相反 (模型特性)
- 对 URDF 模型的关节正方向理解有误

---

## 核心问题总结

### 1. 模型初始状态不稳定

```
现象: 从 z=1.0m 落下后，torso 自然前倾 5-20°
原因: PyBullet humanoid.urdf 的初始质心位置偏前
影响: 需要很大的 ankle torque 才能在短时间内纠正
```

### 2. 欠驱动系统特性

```
自由度: 15个关节 + 6个基座自由度 = 21 DOF
驱动: 只有关节可驱动，基座浮动
问题: 基座姿态只能通过脚-地接触力间接控制
类比: 类似倒立摆，但接触点会滑动/分离
```

### 3. 接触动力学复杂性

```
难点:
- 双脚与地面的接触点不固定
- 摩擦系数影响脚是否滑动
- 单脚/双脚支撑切换

对比: Cart-Pole 的接触是固定的铰链，更简单
```

### 4. 控制策略设计困难

```
多目标冲突:
- 保持高度 ↔ 保持姿态 (膝盖弯曲会影响两者)
- 纠正倾斜 ↔ 保持平衡 (快速纠正可能导致振荡)
- 双脚协调 ↔ 单脚调整 (需要同步或合理分配)
```

## 关键发现

### 发现 1: Position Control vs Torque Control

| 模式 | 优点 | 缺点 |
|------|------|------|
| Position Control | 稳定，不易NaN | 难处理接触力约束 |
| Torque Control | 物理真实，灵活 | 需要精确建模，易不稳定 |

**结论**: 对于站立任务，Position Control 更合适，但需要正确设置目标角度。

### 发现 2: 初始姿态至关重要

```python
# 设置初始姿态
standing_pose = {
    r_hip: -0.1, r_knee: 0.2, r_ankle: -0.1,
    l_hip: -0.1, l_knee: 0.2, l_ankle: -0.1,
}
```

**问题**: 即使设置初始姿态，物理仿真会立即改变它，因为重力和接触力。

### 发现 3: 单关节控制 vs 协调控制

- 只控制膝盖: 高度变化但姿态失控
- 只控制脚踝: 姿态变化但高度失控
- 协调控制: 需要精确的数学模型

## 下一步建议

### 方案 A: 简化到 2D 平面 (推荐)

**理由**:
- 减少自由度 (从 3D 到 sagittal 平面)
- 更容易理解和调试
- 仍能学到核心控制概念

**实现**:
- 约束机器人只在 X-Z 平面运动
- 或使用 2D 双足 URDF (如 Box2D/PyBullet 的 2D 环境)

### 方案 B: 使用参考轨迹

**方法**:
- 预计算一个稳定的站立轨迹
- 使用 PD 控制跟踪轨迹
- 参考: PyBullet 的 `humanoid.py` 示例

### 方案 C: 切换到 Cart-Pole

**理由**:
- 更符合学习路径 (阶段2就是 Cart-Pole)
- 控制原理相同 (倒立摆)
- 更容易成功，建立信心

**方法**:
- 使用 `pybullet_envs` 的 CartPole
- 或自建 URDF

### 方案 D: 研究官方示例

**资源**:
- PyBullet 安装目录下的 `examples/humanoid.py`
- 使用 Gym 环境的 `HumanoidBulletEnv`
