# 双足站立控制 - 经验教训记录

**日期**: 2026-04-06  
**项目**: Phase 1.1 - PyBullet 2D双足站稳  
**状态**: 进行中 - 控制策略需改进

---

## 项目目标

让双足机器人在PyBullet仿真中稳定站立5秒以上，能抵抗小扰动。

---

## 尝试过的控制策略

### 1. 基础PD关节控制 (biped_balance.py)

**方法**:
- 为每个关节独立实现PD控制器
- 根据 torso 倾斜角度调整目标关节角度
- 使用 Torque Control 模式

**结果**: ❌ 失败
- 机器人在 2.7秒 左右摔倒
- 仅控制关节角度，没有考虑整体质心位置

**问题分析**:
- 没有显式控制质心(CoM)在支撑多边形内
- 各关节控制独立，缺乏协调
- 脚踝力矩不足以快速纠正倾斜

---

### 2. 分层控制架构 (biped_balance_v2.py)

**方法**:
- 分层控制: 质心控制 → 关节分配
- 高度控制: 通过膝盖屈伸保持质心高度
- 姿态控制: 通过踝关节调整保持直立
- 使用 Position Control 模式

**结果**: ❌ 数值不稳定 (NaN)
- 控制增益设置过高导致数值溢出
- 即使降低增益，机器人仍有前倾趋势

**问题分析**:
- Position Control 的 `positionGain` 参数与PD增益混淆
- 初始姿态不佳 (落地后 torso 有 5-20° 前倾)
- 高度控制和姿态控制耦合导致冲突

---

### 3. 简化PD控制 (biped_balance_simple.py)

**方法**:
- 简化为三个主要控制量:
  - 高度 → 膝盖弯曲
  - 姿态 → 脚踝角度
  - 位置 → 髋关节补偿
- 使用保守的PD增益

**结果**: ❌ 数值不稳定
- 即使增益很低，仍在 1秒内出现 NaN
- PyBullet 的数值积分器在特定条件下不稳定

**问题分析**:
- Humanoid 模型的关节约束和惯性参数导致数值敏感
- 脚与地面接触时的冲击力未被妥善处理

---

### 4. 纯力矩控制 (biped_balance_stable.py)

**方法**:
- 使用 Torque Control 模式
- 显式设置物理引擎参数 (numSolverIterations, numSubSteps)
- 设置初始站立姿态 (resetJointState)

**结果**: ⚠️ 部分有效但不稳定
- 落地后高度约 0.82m
- 但 torso 仍有 20° 左右倾斜
- 控制启动后 0.4秒 内摔倒

**问题分析**:
- 初始姿态设置后，物理仿真会立即改变关节角度
- 力矩控制响应慢，无法快速纠正大角度误差
- 缺乏对关节角速度的阻尼

---

### 5. 协调关节控制 (biped_balance_working.py)

**方法**:
- 同时控制 hip, knee, ankle 协调运动
- 姿态控制优先 (脚踝纠正倾斜)
- 高度控制限制范围 (避免过度弯曲)

**结果**: ❌ 机器人"起飞"
- 高度从 0.88m 增加到 2.7m
- 膝盖控制逻辑错误导致向上加速

**问题分析**:
- 高度误差计算: `h_err = target - current`
  - 当 height < target 时，h_err > 0
  - 膝盖弯曲为正角度，应该产生向上的力
  - 但实际效果相反 (模型特性)
- 对 URDF 模型的关节正方向理解有误

---

## 核心问题总结

### 1. 模型初始状态不稳定

```
现象: 从 z=1.0m 落下后，torso 自然前倾 5-20°
原因: PyBullet humanoid.urdf 的初始质心位置偏前
影响: 需要很大的 ankle torque 才能在短时间内纠正
```

### 2. 欠驱动系统特性

```
自由度: 15个关节 + 6个基座自由度 = 21 DOF
驱动: 只有关节可驱动，基座浮动
问题: 基座姿态只能通过脚-地接触力间接控制
类比: 类似倒立摆，但接触点会滑动/分离
```

### 3. 接触动力学复杂性

```
难点:
- 双脚与地面的接触点不固定
- 摩擦系数影响脚是否滑动
- 单脚/双脚支撑切换

对比: Cart-Pole 的接触是固定的铰链，更简单
```

### 4. 控制策略设计困难

```
多目标冲突:
- 保持高度 ↔ 保持姿态 (膝盖弯曲会影响两者)
- 纠正倾斜 ↔ 保持平衡 (快速纠正可能导致振荡)
- 双脚协调 ↔ 单脚调整 (需要同步或合理分配)
```

## 关键发现

### 发现 1: Position Control vs Torque Control

| 模式 | 优点 | 缺点 |
|------|------|------|
| Position Control | 稳定，不易NaN | 难处理接触力约束 |
| Torque Control | 物理真实，灵活 | 需要精确建模，易不稳定 |

**结论**: 对于站立任务，Position Control 更合适，但需要正确设置目标角度。

### 发现 2: 初始姿态至关重要

```python
# 设置初始姿态
standing_pose = {
    r_hip: -0.1, r_knee: 0.2, r_ankle: -0.1,
    l_hip: -0.1, l_knee: 0.2, l_ankle: -0.1,
}
```

**问题**: 即使设置初始姿态，物理仿真会立即改变它，因为重力和接触力。

### 发现 3: 单关节控制 vs 协调控制

- 只控制膝盖: 高度变化但姿态失控
- 只控制脚踝: 姿态变化但高度失控
- 协调控制: 需要精确的数学模型

## 新增实验记录 (2026-04-07)

### 本次调试目标
让机器人稳定站立10秒，解决数值不稳定(NaN)问题。

### 关键发现

#### 发现 1: NaN 的根源 - 预稳定阶段

**现象**: 先运行无控制仿真(`stepSimulation`)再启用`setJointMotorControl2`会导致NaN

**测试验证**:
```
测试1: 自稳定后高度 z=0.882m
      然后启用关节控制 -> 立即NaN

测试2: 加载后立即设置关节控制
      无NaN，高度稳定在0.9m
```

**结论**: 必须**立即**设置关节控制，不能先运行无控制仿真。

#### 发现 2: 控制增益的两难

| 增益设置 | 结果 |
|---------|------|
| Kp=0.8, Kd=0.2 (保守) | 1.25秒摔倒，无法纠正前倾 |
| Kp=2.5, Kd=0.8 (中等) | 1.26秒摔倒，略有改善 |
| Kp=3.0, Kd=1.0 (积极) | 1.43秒"起飞"到1.5m |
| positionGain=100 (高) | 立即"起飞"到15m+ |

**核心问题**: 增益太低无法纠正倾斜，增益太高导致数值发散（机器人"起飞"）。

#### 发现 3: 关节方向与直觉相反

**现象**: 膝盖正角度(弯曲)导致机器人向上加速"起飞"

**分析**: 
- 预期: 膝盖弯曲应该降低高度
- 实际: 膝盖弯曲产生向上的力（模型特性与假设不符）

**结论**: 需要重新理解URDF模型的关节正方向。

#### 发现 4: 脚踝控制的局限性

**测试**: 仅使用脚踝控制姿态，保持髋和膝固定

**结果**: 
- 初始0.2°倾斜在1秒内增加到47°
- 脚踝力矩不足以维持直立

**结论**: 仅靠脚踝无法控制15DOF人形机器人的平衡。

### 尝试过的方案

| 方案 | 参数 | 结果 | 原因 |
|------|------|------|------|
| 纯姿态控制 | Kp_p=1.0 | 0.97秒摔倒 | 增益太低 |
| 纯姿态控制 | Kp_p=3.0 | 1.43秒"起飞" | 增益太高 |
| 高度+姿态 | Kp_h=0.5, Kp_p=0.8 | 1.25秒摔倒 | 高度控制冲突 |
| 高PD增益 | positionGain=100 | 立即"起飞" | 数值发散 |
| 保守PD增益 | positionGain=10 | 0.97秒摔倒 | 无法纠正倾斜 |
| 蹲姿初始 | knee=0.3 | 1.2秒"起飞" | 膝盖弯曲方向问题 |
| 直立初始 | knee=0.0 | 立即摔倒 | 质心过高 |
| 渐进控制切换 | 10步渐变 | 0.4秒摔倒 | 切换过程失稳 |
| 无控制预稳定 | 100步 | 立即NaN | 根本原因 |

### 理论反思

根据本次调试，双足站立比预期困难的原因：

1. **模型复杂性**: humanoid.urdf 有15个关节，质量分布不均
2. **接触动力学**: 脚与地面的接触不稳定，容易滑动
3. **控制耦合**: 高度控制和姿态控制通过同一关节（膝盖）实现，互相干扰
4. **数值敏感性**: PyBullet对此模型的积分器在特定条件下不稳定

### 下一步建议

基于MIT 6.832课程的建议，应该：

1. **简化模型**: 先用2D倒立摆(Cart-Pole)验证控制逻辑
2. **固定基座**: 先用固定脚踝的模型测试控制器
3. **ZMP可视化**: 实现ZMP计算，可视化稳定性边界
4. **参考官方**: 研究PyBullet的`humanoid.py`示例的实现方式
