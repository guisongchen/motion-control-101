# PyBullet Stand vs Walk Simulation Comparison

This note compares:

- `biped_walker/sim_stand_balance.py`
- `biped_walker/sim_walker.py`

The goal is to separate the **shared PyBullet simulation scaffolding** from the **task-specific control logic**.

That separation is useful because most future work should focus on the uncommon parts, not on repeatedly rewriting the same setup routines.

## Big picture

Both scripts follow the same high-level simulation shape:

1. prepare or generate the robot URDF
2. start a PyBullet world
3. load the ground and the robot
4. define a motion target for the robot
5. convert that target into robot joint or pose commands
6. step the simulation
7. read back state and contact information
8. compute balance-related metrics such as ZMP
9. plot and summarize the results

So the large-scale PyBullet workflow is already common between them.

The main difference is **what motion is being asked for** and **how that motion is converted into commands**.

## Quick comparison table

| Aspect | `sim_stand_balance.py` | `sim_walker.py` |
|---|---|---|
| Main goal | Stand up and hold upright balance | Follow a multi-step walking pattern |
| Motion source | Hand-defined pelvis trajectory | `build_full_gait(...)` walking pattern |
| Main command target | Joint position targets | Base pose + foot targets + IK result |
| Actuation style | `POSITION_CONTROL` motors | Kinematic reset with IK + `resetJointState` |
| Base behavior | Base evolves under physics | Base is reset to the CoM reference each step |
| Support pattern | Fixed double support | Alternating support and swing |
| IK method | Custom analytic planar leg IK | PyBullet `calculateInverseKinematics2(...)` |
| Main outputs | Pelvis height, tilt, ZMP, force | CoM/ZMP tracking, foot paths, support pattern |
| Main failure concern | Falling or losing upright posture | Gait tracking and IK continuity |

## Shared skeleton table

| Shared layer | Present in both scripts |
|---|---|
| Robot model | Same simplified biped from `create_biped_urdf(...)` |
| Simulator setup | PyBullet connection, gravity, ground plane, URDF loading |
| Runtime pattern | Timestep loop with command -> step -> measure -> log |
| Contact analysis | ZMP-like computation from ground contacts |
| Post-processing | Matplotlib plots and console summaries |

## Focus areas table

| If you want to improve... | Focus here |
|---|---|
| Stand-up quality | Pelvis reference design, joint target generation, posture stabilization |
| Walking quality | Gait generation, footstep timing, swing-foot trajectory, CoM/ZMP planning |
| Physical realism | Reduce kinematic resets, rely more on dynamic actuation and feedback |
| Balance robustness | Add CoM/ZMP feedback, pelvis stabilization, recovery logic |
| Reusability | Extract common PyBullet setup and logging utilities |

## Common parts

## 1. Shared purpose

Both scripts use PyBullet to test ideas on the same simplified biped model.

They are both simulation-side experiments for:

- checking whether a reference motion is physically plausible
- visualizing body motion and contact behavior
- measuring balance-related quantities after the run

## 2. Shared robot model path

Both scripts use the simplified biped generated by:

- `create_biped_urdf(...)`

So they are built on the same kinematic and geometric assumptions about:

- pelvis
- legs
- feet
- ankle height offset

That means comparisons between the two scripts are meaningful, because they are not changing the robot model at the same time.

## 3. Shared PyBullet setup pattern

Both scripts do the usual PyBullet setup steps:

- connect to PyBullet in GUI or DIRECT mode
- add `pybullet_data`
- set gravity
- load `plane.urdf`
- load the biped URDF

This is the reusable simulation boilerplate.

## 4. Shared simulation loop structure

Both scripts run a timestep loop that does some version of:

1. compute a desired robot target
2. send that target into the robot model
3. call `p.stepSimulation()`
4. read robot state
5. read contact information
6. log data

This is the core control/simulation pattern in PyBullet.

## 5. Shared contact-based balance observation

Both scripts estimate a ZMP-like point from robot-ground contacts.

In both cases, the calculation is based on:

- contact location
- normal force
- force-weighted averaging

So both scripts already share the idea that contact data is a key output of the simulator, not just the body pose.

## 6. Shared post-processing style

Both scripts:

- store arrays over time
- generate matplotlib figures
- print summary values to the console

That means the evaluation pattern is also shared:

- run the simulation
- inspect trajectory tracking
- inspect balance/contact behavior
- summarize key errors or stability indicators

## Uncommon parts

## 1. Different task goals

### `sim_stand_balance.py`

Goal:

- stand up from a crouched pose
- remain upright in place

The task is mostly about:

- posture generation
- balance while staying in double support
- maintaining a stable upright stance

### `sim_walker.py`

Goal:

- follow a walking pattern over multiple steps

The task is mostly about:

- gait generation
- footstep timing
- swing/support alternation
- CoM/ZMP tracking through a walking sequence

This is the first big split: **standing** versus **locomotion**.

## 2. Different motion sources

### `sim_stand_balance.py`

The reference motion is simple and local:

- crouch pelvis target
- smooth upward pelvis transition
- standing pelvis target

This comes from:

- `desired_pelvis_position(sim_time)`

So the motion source is a hand-defined stand-up trajectory.

### `sim_walker.py`

The reference motion is generated externally by a walking pattern generator:

- `build_full_gait(...)`

This produces:

- CoM reference
- ZMP reference
- left foot trajectory
- right foot trajectory
- support phase information

So the motion source is a much richer gait-planning pipeline.

This is a major place to focus future work: **motion generation logic** is much more central in walking than in standing.

## 3. Different command layers

### `sim_stand_balance.py`

The script computes joint targets from pelvis geometry and sends them through:

- `p.setJointMotorControl2(..., POSITION_CONTROL, ...)`

So it is using PyBullet motor control as an actuator layer.

That makes it closer to:

- a simple controller-driven dynamic simulation

### `sim_walker.py`

The script computes foot target positions, solves IK, and then writes states with:

- `p.resetBasePositionAndOrientation(...)`
- `p.resetJointState(...)`

So it is much more kinematic in character.

That makes it closer to:

- trajectory playback inside a physics engine

This is one of the most important differences in the whole comparison.

## 4. Different inverse kinematics strategy

### `sim_stand_balance.py`

Uses a custom analytic leg solution:

- `solve_pitch_leg_angles(...)`

Characteristics:

- planar
- symmetric
- simple
- specialized to this stand-up problem

### `sim_walker.py`

Uses PyBullet's built-in:

- `calculateInverseKinematics2(...)`

Characteristics:

- multi-end-effector
- both feet solved together
- more general
- better suited for moving foot targets during gait

So the IK problem itself is different:

- standing uses a constrained symmetric leg geometry problem
- walking uses a general moving-foot IK problem

## 5. Different use of the floating base

### `sim_stand_balance.py`

The robot base is loaded once and then allowed to evolve under physics.

The code reads:

- actual base position
- actual base orientation

as outputs of the simulation.

That means the simulator decides how the body moves after joint commands are applied.

### `sim_walker.py`

The robot base pose is reset every timestep to the desired CoM reference.

That means the base trajectory is largely imposed from outside rather than emerging from dynamics.

This is a key modeling difference:

- `sim_stand_balance.py` asks: **can the robot remain upright under this posture control?**
- `sim_walker.py` asks: **what happens if I replay this walking pattern kinematically in PyBullet?**

## 6. Different contact/support assumptions

### `sim_stand_balance.py`

Assumes:

- double support
- both feet stay on the ground
- support region is stationary

So balance is interpreted around a fixed support arrangement.

### `sim_walker.py`

Assumes:

- alternating support feet
- moving footsteps
- swinging foot trajectories
- changing support geometry over time

So contact timing and support transitions are part of the problem itself.

This makes walking much more than "standing plus forward motion."

## 7. Different error-handling/failure logic

### `sim_stand_balance.py`

Has an explicit failure rule:

- if base height drops below `0.35 m`, the robot is considered to have lost balance

Then it:

- truncates logs
- exits early
- reports failure timing

### `sim_walker.py`

Does not use the same kind of fall detector.

Instead it mainly handles:

- IK failures by reusing the previous solution

So the failure model is different:

- standing watches for physical collapse
- walking watches more for kinematic solve continuity

## 8. Different evaluation outputs

### `sim_stand_balance.py`

Focuses on:

- pelvis height tracking
- tilt angles
- ZMP inside support area
- total ground reaction force

These are metrics for posture stability.

### `sim_walker.py`

Focuses on:

- CoM reference vs actual
- ZMP reference vs actual
- foot trajectories
- support foot locations
- CoM and ZMP RMS tracking errors

These are metrics for gait tracking and walking pattern quality.

## Practical takeaway

If the goal is to understand the **big picture of simulation in PyBullet**, the shared structure is already clear:

- world setup
- robot loading
- timestep loop
- contact/state readback
- plotting and summary

That part is mostly reusable infrastructure.

If the goal is to decide **where to focus engineering effort**, the uncommon parts matter much more.

## Where to focus next

Instead of spending most time on PyBullet setup routines, the better focus is usually here:

### 1. Motion generation

For standing:

- how the pelvis target should move
- whether posture references should include tilt or lateral correction

For walking:

- step timing
- CoM planning
- ZMP planning
- foot swing path design

## 2. Command representation

Decide what layer should command the robot:

- direct joint targets
- analytic IK outputs
- PyBullet IK outputs
- torque control
- whole-body control

This choice changes the meaning of the simulation much more than changing small setup details.

## 3. Dynamic realism versus kinematic replay

Right now, the two scripts sit at different points on that spectrum:

- `sim_stand_balance.py` is more dynamic
- `sim_walker.py` is more kinematic

So an important design question is:

- do you want walking to emerge more from dynamics?
- or do you mainly want to visualize and validate planned trajectories?

That decision should guide future refactoring.

## 4. Balance control strategy

The current standing script is mostly posture holding.

The current walking script mainly replays a pattern.

If you want stronger balance behavior, future work should likely go into:

- CoM/ZMP feedback
- pelvis stabilization
- foot placement adaptation
- disturbance rejection
- better support-phase logic

## 5. Contact modeling and support logic

For standing, the important issue is:

- how to keep the support point centered and stable

For walking, the important issue is:

- when and where support shifts between feet

That means contact planning becomes much more important once the task moves from standing to walking.

## Short conclusion

The common parts show the **PyBullet simulation skeleton**.

The uncommon parts show the **actual robotics problem** being studied.

So if you already understand the shared setup, the next work should mostly focus on:

1. trajectory generation
2. IK/control method
3. base motion treatment
4. contact/support transitions
5. balance feedback design

Those are the parts that determine whether the simulator is just replaying motion, holding posture, or becoming a more realistic locomotion controller testbed.
